\section{What Is Zero Trust Architecture?}

\subsection{The Core Idea in Plain Language}
% Zero Trust begins with a single premise: do not trust any user,
% device, or system by default — regardless of where they are.
% Every access request must be authenticated, authorized, and
% continuously validated. Trust is earned per-request, not granted at the door.
%
% Use the "badge scan on every door" analogy here — even if you
% already badged into the building, each room requires a new scan.

\subsection{Origin and Formalization}
% John Kindervag (Forrester, 2010) coined the term and the framework.
% Google built it internally as BeyondCorp after the 2009 Operation Aurora breach.
% NIST SP 800-207 (2020) formalized ZTA as a national standard.
% Executive Order 14028 (2021) made it a federal mandate.

\subsection{The Three Foundational Principles}

    \subsubsection{Never Trust, Always Verify}
    % Every request — from inside or outside the network — is treated
    % as potentially hostile until proven otherwise. Authentication is
    % continuous, not a one-time event at login.

    \subsubsection{Least-Privilege Access}
    % Users, devices, and applications receive only the permissions
    % required to perform their specific function. Nothing more.
    % This limits how far an attacker can move even with valid credentials.

    \subsubsection{Assume Breach}
    % The system is designed on the assumption that a breach will occur —
    % or already has. The goal is containment and rapid detection,
    % not just prevention. This flips the entire defensive posture.

\subsection{What ZTA Is Not}
% ZTA is not a product, not a single technology, and not a one-time project.
% It is a security philosophy and architectural strategy implemented
% across an organization's people, processes, and technology over time.

% IV. HOW IT WORKS: THE TECHNICAL ARCHITECTURE  (~1.5 pages)
% Tone: Technical — this is the "how" section.
% Audience: Informed reader / CS/IT student level.
% ══════════════════════════════════════════════════════════════════════════════
\section{How Zero Trust Works: Technical Architecture and Components}

\subsection{The NIST Logical Architecture}
% Describe the three-component model from NIST SP 800-207:
% - Policy Engine (PE): makes the trust decision
% - Policy Administrator (PA): communicates the decision
% - Policy Enforcement Point (PEP): executes the decision
% Walk through how a single access request flows through all three.

\subsection{Identity as the New Perimeter}

    \subsubsection{Identity and Access Management (IAM)}
    % In ZTA, identity replaces the network as the primary trust boundary.
    % Who you are and what you're permitted to do drives every decision.

    \subsubsection{Multi-Factor Authentication (MFA)}
    % Passwords alone are insufficient. MFA requires a second proof:
    % hardware token, authenticator app, biometric, etc.

    \subsubsection{Single Sign-On and Identity Providers}
    % Centralized platforms (Okta, Azure Active Directory, Google Workspace)
    % act as the authoritative source of identity across all systems.

\subsection{Device Trust}

    \subsubsection{Posture Assessment}
    % Before granting access, the system checks device compliance:
    % OS patch level, encryption status, EDR presence, enrollment status.
    % A device that fails posture checks is denied or sandboxed.

    \subsubsection{Continuous Device Monitoring}
    % Device trust is not checked once at connection — it is re-evaluated
    % continuously. A device that becomes non-compliant mid-session
    % can lose access in real time.

\subsection{Micro-Segmentation}

    \subsubsection{Replacing the Flat Network}
    % Traditional networks are "flat" — once inside, lateral movement is easy.
    % Micro-segmentation divides the network into isolated zones,
    % each requiring separate authorization to enter.

    \subsubsection{Controlling East-West Traffic}
    % Most attacks move laterally (east-west) after initial entry.
    % Micro-segmentation is the primary technical control against this.

\subsection{Continuous Monitoring and Analytics}

    \subsubsection{Real-Time Telemetry}
    % Every action generates logs: logins, file accesses, API calls, anomalies.
    % This data feeds into SIEM systems and behavioral analytics platforms.

    \subsubsection{Adaptive Policy Enforcement}
    % Risk scores update continuously. A user flagged mid-session
    % can be step-up authenticated, rate-limited, or terminated automatically.
    % Machine learning models identify anomalies faster than humans can.

\subsection{Data Protection}

    \subsubsection{Encryption Everywhere}
    % mTLS (mutual TLS) ensures both ends of every connection are verified.
    % Data at rest is encrypted; data in transit is encrypted.

    \subsubsection{Data Loss Prevention (DLP)}
    % Even authorized users can trigger DLP policies if their behavior
    % looks like exfiltration — unusual volumes, unusual destinations, etc.

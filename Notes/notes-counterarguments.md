## Checklist (for each: acknowledge → counter)

### "It's too expensive and complex"
- [ ] Acknowledge: true, especially for legacy-heavy organizations; this is a real barrier
- [ ] Counter: cost of a breach (IBM report number) vs. cost of ZTA implementation;
      the question is not whether to afford ZTA, but whether to afford a breach

### "It creates too much user friction"
- [ ] Acknowledge: MFA fatigue is real; push bombing attacks exploit it
- [ ] Counter: modern risk-based authentication = friction only when behavior is anomalous;
      seamless for normal use. Good UX and good security are not mutually exclusive.

### "ZTA doesn't stop everything"
- [ ] Acknowledge: true — insider threats with valid credentials, zero-days, social engineering
      all exist outside ZTA's control
- [ ] Counter: no security model eliminates all risk. ZTA "fails better" than the alternative.
      A contained breach vs. an uncontrolled one is an enormous difference.

### Vendor lock-in
- [ ] Acknowledge: real concern in multi-vendor / multi-cloud environments
- [ ] Counter: open standards (NIST SP 800-207, FIDO2, OpenID Connect) exist to prevent this;
      responsible implementations use them
